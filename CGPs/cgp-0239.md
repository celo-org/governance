## Overview

This proposal activates nine additional validator groups in the stCELO `DefaultStrategy`, expanding the active set from 6 to 15. The protocol-wide receivable capacity grows from approximately **0.6M CELO to ~20M CELO** (~30×), restoring meaningful inflow headroom for stCELO and reducing the chance that new deposits get throttled by groups near their elected-vote caps.

### Proposal Description

stCELO's `DefaultStrategy` distributes incoming votes across an ordered set of "active" validator groups. Each group has a maximum vote capacity (per the Celo Election contract), and once that capacity is exhausted the protocol must rely on the remaining groups to absorb new deposits. With only 6 active groups today, available headroom is small relative to the broader Celo staking landscape and concentrates protocol exposure on a handful of operators.

Selection of the nine candidate groups follows a transparent scoring formula — `score = 0.7 × headroom + 0.3 × cap` (normalized to the candidate-set maximum) — and an on-chain eligibility filter (`GroupHealth.isGroupValid(group)`). All nine candidates currently return `isGroupValid = true` on Celo mainnet (verified 2026-05-18). The selected nine, in headroom order, are:

### Mechanism

`DefaultStrategy.activateGroup(group, lesser, greater)` requires the group to first be present in the contract's `activatableGroups` set. Moving a group into `activatableGroups` is done via `addActivatableGroup(address)`, which is `onlyOwner`. Once a group is activatable and `GroupHealth.isGroupValid(group) == true`, anyone can call `activateGroup` and the contract inserts the group into the sorted active list (with zero initial votes, so the new entries go at the lowest-votes end of the sorted set — `lesser = 0x0`, `greater = current tail`).

So the proposal naturally splits into two phases:

1. **Owner-only:** nine `addActivatableGroup(group)` calls on `DefaultStrategy`.
2. **Permissionless follow-up:** nine `activateGroup(group, lesser, greater)` calls — can be triggered by any caller after phase 1 lands.

### Authorization and execution model

`DefaultStrategy` (proxy `0x3A3ed74B1cC543D5EB323f70ac2F19977a0eA088`) is owned by the **StakedCelo MultiSig** (`0x78DaA21FcE4D30E74fF745Da3204764a0ad40179`), but that MultiSig exposes a dedicated entrypoint for Celo Governance:

```solidity
// staked-celo/contracts/common/MultiSig.sol
function governanceProposeAndExecute(
    address[] calldata destinations,
    uint256[] calldata values,
    bytes[] calldata payloads
) external onlyGovernance {
    for (uint256 i = 0; i < destinations.length; i++) {
        bytes memory returnData = ExternalCall.execute(destinations[i], values[i], payloads[i]);
        emit GovernanceTransactionExecuted(i, returnData);
    }
}
```

`onlyGovernance` checks `msg.sender == address(getGovernance())`, where `getGovernance()` resolves to the Celo Governance proxy at `0xD533Ca259b330c7A88f74E000a3FaEa2d63B7972`. When Celo Governance invokes `governanceProposeAndExecute`, the MultiSig itself becomes the `msg.sender` of each forwarded inner call — and since the MultiSig is `DefaultStrategy.owner()`, the `onlyOwner` check on `addActivatableGroup` passes naturally. **This CGP therefore self-executes through Celo Governance.** No multisig signer coordination, no 7-day multisig delay.

## Proposed Changes

### Transaction Descriptions

A single Celo Governance transaction:

- Destination: `StakedCelo MultiSig` (`0x78DaA21FcE4D30E74fF745Da3204764a0ad40179`)
- Function: `governanceProposeAndExecute(address[] destinations, uint256[] values, bytes[] payloads)`
- Arguments:
  - `destinations[]` — nine entries, each `0x3A3ed74B1cC543D5EB323f70ac2F19977a0eA088` (`DefaultStrategy`)
  - `values[]` — nine entries, each `0`
  - `payloads[]` — nine ABI-encoded calldata blobs for `DefaultStrategy.addActivatableGroup(address group)` (selector `0xb95de2d6`), one per candidate group, in the order listed in the table above
- Value: 0

Each inner payload places one of the nine validator groups into `DefaultStrategy.activatableGroups`. Once this proposal executes, the permissionless follow-up step — anyone calling `DefaultStrategy.activateGroup(group, lesser, greater)` — moves each group into the active sorted set with zero initial votes (`lesser = 0x0`, `greater = current tail`). The permissionless follow-up is not part of this CGP and is not gated by any governance action.

### Json Script

See [mainnet.json](cgp-0239/mainnet.json). It contains the single `governanceProposeAndExecute` transaction with the three array arguments fully expanded.

### Fork test

The calldata was validated on a Celo mainnet anvil fork by impersonating the Celo Governance proxy (`0xD533Ca25...`), calling `MultiSig.governanceProposeAndExecute(destinations, values, payloads)` with the contents of `mainnet.json`, and then running the nine permissionless `activateGroup` follow-ups. Post-execution assertions confirmed `DefaultStrategy.getNumberOfGroups() == 15` (was 6), `DefaultStrategy.isActive(group) == true` for each of the nine new groups, and `DefaultStrategy.activatableGroupsCount() == 0`.
