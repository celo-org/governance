## 1. Summary

This proposal requests funding to compensate five independent infrastructure operators: **@swiftstaking, @kamikazechaser, @thylacine, @clemens, @dee** for running **Celo Challengers** and the supporting infrastructure required to do so reliably over a 7-month period.

Running a challenger requires operators to maintain the following infrastructure:

* **full Ethereum/L1 node stack or equivalent trusted L1 connectivity**
* **Celo L2 archive node**
* **challenger service itself**, with local derivation and historical state access

The Celo challenger documentation also notes that challenger operators should run as much of the stack locally as possible, that historical state roots may need to be checked up to roughly one week back, and that an **archive node** and **dedicated EigenDA proxy** are part of a robust setup.

The goal of this proposal is to strengthen Celo's fault-proof and dispute infrastructure by ensuring that multiple independent operators have resources to run challengers continuously, monitor network behavior, and remain ready to challenge invalid state proposals when necessary.

---

## 2. Motivation

Celo's security model benefits from having multiple independent challenger operators actively running the infrastructure required to verify proposed state roots and react if an invalid proposal appears. The challenger compares the proposed dispute game L2 state root against the operator's own locally derived state, and when there is a mismatch, the challenger can publish a challenge on L1.

That means challenger operations impose real and ongoing infrastructure costs. Operators must maintain a properly configured **Celo archive node in full-sync mode**, ensure access to the required L1 data sources, and operate the challenger service continuously.

This compensation helps ensure sustained uptime, distributed responsibility and continuity of operations.

---

## 3. Specification

This proposal funds five independent operators to run Celo challenger infrastructure for a period of seven months.

Each funded operator will be expected to:

1. Run and maintain the infrastructure required to support a Celo Challenger.
2. Respond to incidents, configuration updates, software upgrades, and operational issues during the funding period.

Challenger needs to compare proposed state roots against a synced L2 state. because historical state roots may need to be checked for approximately a week, the operator must run a local **L2 archive node**. The setup also calls for a dedicated **EigenDA proxy**.

Each operator will be independently responsible for their own infrastructure, monitoring, and maintenance.

## 4. Metrics and KPIs

Success for this initiative will be evaluated using the following metrics:

* **Operator continuity:** 5 funded operators remain active until the end of term.
* **Infrastructure availability:** challenger and related services remain operational with minimal downtime.
* **Operational readiness:** operators maintain configurations required to actively monitor and, where applicable, challenge invalid proposals.
