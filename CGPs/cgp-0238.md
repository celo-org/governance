## Overview
 
In this governance proposal, we propose to execute the Celo Core Contracts Release 16. This release follows the standard [release process](https://docs.celo.org/community/release-process/smart-contracts).

Changes that are being proposed are described in [this Github release](https://github.com/celo-org/celo-monorepo/releases/tag/core-contracts.v16). Trails of Bits has completed a code audit of this release, and the tooling provided ensures the version control references included in the report match the contract deployment described in the proposal content.

## Transactions additional to contract changes

This is the first contract release where we support submitting and verifying transactions appended to the Contracts Release. These transactions are meant to retroactively fund the LockedCelo contract. This change does not represent a change to the current release schedule.

Because `CeloUnreleasedTreasury.release` may only be called by the `EpochManager`, the registry entry for `EpochManager` is temporarily pointed to the `Governance` contract for the duration of the proposal and restored at the end.

1. Contract updates, in this case only setting a new implementation to `EpochManager`. This is auto-generated by the tooling and verified by the verification script.

3. Temporarily set `Governance` as `EpochManager` in the `Registry`
  - Destination: `Registry.setAddressFor` at `0x000000000000000000000000000000000000ce10`
  - Data: identifier `"EpochManager"`, address `0xD533Ca259b330c7A88f74E000a3FaEa2d63B7972` (the `Governance` proxy), so that the next transaction passes the `onlyEpochManager` check
  - Value: 0 CELO; this is a registry update and does not transfer funds

4. Release 1.98M CELO from `CeloUnreleasedTreasury` to `LockedCelo`
  - Destination: `CeloUnreleasedTreasury.release`
  - Data: beneficiary `0x6cC083Aed9e3ebe302A6336dBC7c921C9f03349E` (the `LockedCelo` proxy), amount `1979490000000000000000000` (1,979,490 CELO in wei)
  - Value: 0 CELO is sent with the call itself; the 1,979,490 CELO is transferred from `CeloUnreleasedTreasury`'s balance to `LockedCelo` to retroactively fund the contract

5. Restore the original `EpochManager` address in the `Registry`
  - Destination: `Registry.setAddressFor` at `0x000000000000000000000000000000000000ce10`
  - Data: identifier `"EpochManager"`, address `0xF424B5e85B290b66aC20f8A9EAB75E25a526725E` (the original `EpochManager` proxy), reverting the temporary change from transaction 1
  - Value: 0 CELO; this is a registry update and does not transfer funds

 
## Verification

### Smart Contracts

#### Dependencies

1. CeloCLI, can be installed with `$ npm install -g @celo/celocli@latest`
2. Foundry toolset:
   1. Install the version manager with `$ curl -L https://foundry.paradigm.xyz | bash` (after you need to restart your terminal or run `source ~/.bashrc / source ~/.zshrc`)
   2. Install foundry itself with `foundryup`


#### Code to verify
 
Make sure to download latest [celo-monorepo](https://github.com/celo-org/celo-monorepo/) with

`$ git clone git@github.com:celo-org/celo-monorepo.git --branch release/core-contracts/16`

And a version of the cli compatible with this contract release, you can install it with:

#### Build and verify

Then build the project with:

`$ yarn && yarn build`

Once built, `cd` to the protocol folder with:

`$ cd packages/protocol`

From there, you can download the current proposal:

```bash
$ celocli governance:show --proposalID TODO --jsonTransactions release-16-proposal.json --node https://forno.celo.org
```

And then verify that the proposal deployment matches the audited release:

```
$ yarn release:verify-deployed:foundry -p release-16-proposal.json -b core-contracts.v16 -n mainnet -f -i "releaseData/initializationData/release16.json" -e releaseData/extraTransactions/release16-mainnet.json
```

The output should verify that the bytecodes and storages match what is expected, it should look something like this:

```
Success, no bytecode mismatches found!
Writing linked library addresses to libraries.json
✨  Done in 30.86s.
```

### Retroactive funding LockedCelo contract

The full calculation below is automated in [`cgp-0238/calculate-locked-celo-shortfall.sh`](./cgp-0238/calculate-locked-celo-shortfall.sh). Run it to reproduce the numbers against current chain state — it will prompt for the pending-withdrawals value (linked Dune query) and print the shortfall and projected funding amount.

Get the Celo contract address:

`$ LOCKEDCELO_CONTRACT_ADDRESS=$(celocli network:contracts -n https://forno.celo.org --filter='Contract=LockedCelo' --output json 2>/dev/null | jq -r '.[0].proxy')`

Get the Celo balance of the contract:

`$ cast balance $LOCKEDCELO_CONTRACT_ADDRESS --rpc-url https://forno.celo.org` = 81,687,800.321286

Query the total amount of Celo locked in the contract:

`$ cast call $LOCKEDCELO_CONTRACT_ADDRESS "getTotalLockedGold()(uint256)" --rpc-url https://forno.celo.org`

At the time of writing LOCKEDCELO_TOTAL_LOCKED = 79,797,423.784188


At the time of writing, LOCKEDCELO_CONTRACT_BALANCE = 81,687,800.321286

Use this [Dune query](https://dune.com/queries/7436638/11378487/?utm_source=share&utm_medium=copy&utm_campaign=visualization) you can query the total pending withdraws.

At the time of writing, LOCKEDCELO_TOTAL_PENDING_WITHDRAWS = 3,832,072.403168

Total shortfal is LOCKEDCELO_CONTRACT_BALANCE - LOCKEDCELO_TOTAL_PENDING_WITHDRAWS - LOCKEDCELO_TOTAL_LOCKED = -1,941,695.866071.

As it's been aprox 411 days since Celo L2 launched (it was March 26th 2025), we add the equivalent shortfall to estimate the amount at the time the proposals passes (roughly 8 days from now) 1 + 8/411.

Total amount to be funded: 1,941,695*(1 + 8/411) = 1,979,490.43

This estimation is not exact, but the potential difference would be negligible at the time of executing.
 
## Risks
 
Celo Core Contracts are critical for the functioning of the Celo Platform. While cLabs have invested significant efforts to expand testing and verification tooling, and a third party audit has been completed, errors in this proposal could lead to situations that are only recoverable with a very difficult hard-fork.
